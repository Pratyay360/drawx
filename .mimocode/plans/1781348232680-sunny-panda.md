# Plan: Migrate from JSON file to libSQL (SQLite) database

## Context
Currently, the Tauri backend stores all data in a single JSON file (`drawx_data.json`) using serde_json. Every operation reads the entire file, mutates it in memory, and writes it back. The user wants to migrate to libSQL (SQLite) for proper database storage.

## Architecture Decision
Use **`rusqlite`** crate in the Rust backend. This is the standard SQLite library for Rust — mature, well-documented, and perfect for local-first Tauri apps. The frontend (`src/services/tauri.ts`) remains unchanged since it already talks to the same Tauri commands.

## Database Schema

```sql
CREATE TABLE IF NOT EXISTS canvases (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    description TEXT,
    elements TEXT NOT NULL DEFAULT '[]',   -- JSON array
    app_state TEXT NOT NULL DEFAULT '{}',  -- JSON object
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

## Files to modify

### 1. `src-tauri/Cargo.toml` — Add rusqlite dependency
```toml
rusqlite = { version = "0.31", features = ["bundled"] }
```

### 2. `src-tauri/src/lib.rs` — Replace JSON with SQLite
- Remove `AppData` struct, `load_data()`, `save_data()`, `data_path()`
- Add `fn init_db(app) -> Connection` — opens/creates DB at `app_data_dir/drawx.db`
- Add helper `fn get_db(app) -> Connection` using Tauri state management
- Rewrite all Tauri commands to use SQL queries:
  - `list_canvases` → `SELECT * FROM canvases ORDER BY updated_at DESC`
  - `create_canvas` → `INSERT INTO canvases ...`
  - `load_canvas` → `SELECT * FROM canvases WHERE id = ?`
  - `save_canvas` → `UPDATE canvases SET elements = ?, app_state = ?, updated_at = ? WHERE id = ?`
  - `delete_canvas` → `DELETE FROM canvases WHERE id = ?`
  - `update_canvas_title` → `UPDATE canvases SET title = ?, updated_at = ? WHERE id = ?`
- Store the `Connection` in Tauri managed state for reuse
- Keep the same command signatures so frontend is unchanged
- Remove unused project-related commands (`list_projects`, `save_project`, `open_project`, `greet`)

### 3. No frontend changes needed
`src/services/tauri.ts` already uses the same Tauri invoke commands and the interface remains identical.

## Implementation Steps

1. Add `rusqlite` to `Cargo.toml`
2. Rewrite `lib.rs`:
   - Initialize SQLite DB with schema on app startup
   - Store `Connection` in Tauri managed state
   - Implement all CRUD commands with SQL
   - Remove JSON file code and unused project commands
3. Remove `drawx_data.json` handling (data migration not needed — fresh start or optional migration)

## Verification
1. `cargo check` in `src-tauri/` to verify Rust compiles
2. `npm run lint` / `npx tsc --noEmit` to verify frontend still compiles
3. Manual test: launch app, create canvases, verify persistence across restarts
